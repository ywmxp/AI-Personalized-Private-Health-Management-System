## 1. 概述

前端作为 **SPA（单页应用）**，不拥有独立的数据库。数据存储分为三个层次：

|层次|技术|生命周期|用途|
|-|-|-|-|
|服务端状态|后端 API + MySQL|持久化|所有业务数据（用户、健康数据、提醒等）|
|客户端缓存|localStorage|持久化（跨会话）|token、用户信息、已读推送 ID|
|运行时状态|Pinia Store（响应式内存）|会话期间|用户登录态、跨组件共享数据|
|内存缓存|JavaScript Map|会话期间|知识条目详情缓存|

```Mermaid

    end

    subgraph "浏览器存储"

    end

    DB <-->|"API 请求"| PINIA
    PINIA <-->|"双向同步"| LS
    PINIA -->|"读取"| REF
    CACHE -->|"读取"| REF
    API <-->|"CRUD"| DB
```

## 2. localStorage 持久化设计

### 2.1 存储项定义

|Key|类型|读写方|说明|
|-|-|-|-|
|`token`|string|userStore, request.ts, router|JWT 认证令牌|
|`userInfo`|JSON string|userStore, AppSidebar, router|用户信息对象（User 类型）|
|`role`|string|router 守卫|用户角色（`admin` 或 `user`），用于管理员权限判断|
|`aihealth_read_pushes`|JSON string (number[])|KnowledgeView|已读推送 ID 数组|

### 2.2 存储结构详述

#### token

```Plain Text
Key: "token"
Value: "eyJhbGciOiJIUzI1NiIs..."  (JWT string)
写入时机: 登录成功后
清除时机: 登出、401 响应
读取者: request.ts 拦截器、router 守卫、userStore 初始化
```

#### userInfo

```TypeScript
// Key: "userInfo"
// Value: JSON.stringify(User)
interface User {
  userId: number
  username: string        // 用户名
  phone: string           // 手机号
  birthDate?: string      // 出生日期 yyyy-MM-dd
  gender?: 'MALE' | 'FEMALE' | 'UNKNOWN'
  height?: number         // 身高 cm
  role: 'USER' | 'ADMIN'  // 角色
}
```

#### role

```Plain Text
Key: "role"
Value: "admin" | "user"
写入时机: 登录成功后（与 userInfo 同时写入）
读取者: router 守卫（admin 路由权限判断）
```

#### aihealth_read_pushes

```TypeScript
// Key: "aihealth_read_pushes"
// Value: "[1, 3, 5, 12]"  (JSON 数组)
const READ_PUSHES_KEY = 'aihealth_read_pushes'

function getReadPushIds(): Set<number> {
  return new Set(JSON.parse(localStorage.getItem(READ_PUSHES_KEY) || '[]'))
}

function saveReadPushId(pushId: number) {
  const ids = getReadPushIds()
  ids.add(pushId)
  localStorage.setItem(READ_PUSHES_KEY, JSON.stringify([...ids]))
}
```

### 3.2 状态初始化流程

```Mermaid
sequenceDiagram
    participant App as App 启动
    participant Store as userStore
    participant LS as localStorage

    App->>Store: createPinia() + app.use(pinia)
    Store->>Store: defineStore('user', ...)
    Store->>LS: localStorage.getItem('token')
    LS-->>Store: token (string | null)
```

**关键设计：**

- `parseStoredUserInfo()` 包裹在 try-catch 中，JSON 解析失败时清除损坏数据

- token 和 userInfo 分开初始化，各自独立容错

### 4.3 Mock 双开关设计

```Plain Text
.env 中的两个独立开关:

VITE_MOCK_API=true    → 健康数据 / 提醒 / 知识库 / 管理后台使用 Mock
VITE_MOCK_AUTH=true   → 登录 / 注册使用 Mock
```

- 两个开关独立控制，互不影响

- 例如可以 `VITE_MOCK_AUTH=true` + `VITE_MOCK_API=false`：登录 Mock 但数据走真实后端

- Mock 响应格式与真实后端完全一致（`{ code: 0, message: 'success', data: ... }`）

## 5. 内存缓存设计

### 5.1 知识缓存

```TypeScript
// KnowledgeView.vue 内部
const knowledgeCache = ref<Map<number, KnowledgeItem>>(new Map())

// 缓存写入：知识库列表加载后
for (const item of knowledgeList.value) {
  knowledgeCache.value.set(item.knowledgeId, item)
}

// 缓存读取：推送详情弹窗
const cached = knowledgeCache.value.get(item.knowledgeId)
```

**目的：** 推送列表只有摘要，点击查看详情时优先从缓存读取完整内容，避免额外请求。

### 5.2 已检查提醒集合

```TypeScript
// ReminderPopup.vue 内部
const checkedReminders = new Set<number>()

// 触发后标记
checkedReminders.add(reminder.reminderId)

// 稍后提醒时清除标记（10 分钟后恢复检查）
setTimeout(() => {
  checkedReminders.delete(reminderId)
}, 10 * 60 * 1000)
```

**目的：** 防止同一提醒在相邻轮询周期中重复弹出。

## 7. 数据安全

|措施|说明|
|-|-|
|Token 携带|通过 Axios 拦截器自动附加 `Authorization: Bearer <token>`|
|Token 过期|后端返回 401 时自动清除本地状态并跳转登录页|
|密码处理|前端不存储密码，仅通过 HTTPS 传输（生产环境）|
|XSS 防护|Vue 默认对模板插值进行 HTML 转义|
|Mock 数据|Mock 数据仅在开发环境使用，生产构建时关闭|
|管理员权限|路由守卫 + 侧边栏双重校验 `role === 'admin'`|

## 8. 设计权衡

### 8.1 用 localStorage 而不是 SessionStorage

- localStorage 跨会话持久化，用户关闭浏览器后重新打开无需重新登录

- JWT token 本身有过期时间（`JWT_EXPIRE_MINUTES=120`），安全性由过期机制保证

### 8.2 token 不存 Pinia 而同时存 localStorage

- Pinia 刷新即丢失 — localStorage 是真实持久化

- 但 Pinia 是响应式的 — 组件可以直接 `userStore.isLoggedIn` 响应登录态变化

- 结论：**双重存储**，localStorage 是数据源，Pinia 是响应式视图

### 8.3 路由守卫读 localStorage 而不读 Pinia

- 路由守卫在 Pinia 完全初始化前就可能执行

- 直接读 localStorage 避免时序依赖
