# 后端接口草案 v1

本文档是前后端联调用的第一版接口契约。接口命名和字段允许在评审后调整，但调整后必须同步更新本文档。

## 1. 通用约定

### 1.1 基础地址

本地开发：

```text
http://127.0.0.1:8080
```

### 1.2 统一响应格式

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

约定：

- `code = 0` 表示成功。
- 非 0 表示失败。
- `message` 给前端展示或调试。
- `data` 返回具体业务数据。

### 1.3 分页响应格式

```json
{
  "items": [],
  "page": 1,
  "size": 10,
  "total": 0
}
```

约定：

- `page` 从 1 开始。
- `size` 默认 10。
- 时间字段统一使用 `yyyy-MM-dd HH:mm:ss`。

### 1.4 登录凭证

登录成功后返回 `token`。除注册、登录外，业务接口默认要求请求头携带：

```http
Authorization: Bearer <token>
```

## 2. 认证与用户接口

### 2.1 用户注册

```http
POST /api/auth/register
```

请求：

```json
{
  "phone": "13800138000",
  "username": "test_user",
  "password": "123456",
  "confirmPassword": "123456"
}
```

响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "userId": 1
  }
}
```

规则：

- 手机号唯一。
- 密码和确认密码必须一致。
- 密码必须加密存储。
- 该接口只用于普通用户注册。
- 请求体不接受 `role` 字段，后端固定创建 `USER` 角色。
- 管理员账号第一阶段通过初始化 SQL 创建，不开放公开注册接口。

## 3. 健康数据接口

### 3.1 新增健康数据

```http
POST /api/health-data
```

请求：

```json
{
  "dataType": "WEIGHT",
  "dataValue": "70.5",
  "unit": "kg",
  "recordTime": "2026-05-24 08:30:00",
  "overwrite": false
}
```

字段说明：

| 字段 | 必填 | 说明 |
|---|---|---|
| `dataType` | 是 | 仅支持 `BLOOD_PRESSURE`、`BLOOD_GLUCOSE`、`WEIGHT`、`EXERCISE_MINUTES`、`SLEEP_HOURS` |
| `dataValue` | 是 | 健康数据值，按字符串传输并由后端按类型校验 |
| `unit` | 否 | 单位；为空时由后端按类型补默认单位 |
| `recordTime` | 否 | 记录时间，格式固定为 `yyyy-MM-dd HH:mm:ss`；为空时由后端默认写入当前时间 |
| `overwrite` | 否 | 是否覆盖同时间同类型已有记录，默认 `false` |

响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "dataId": 1
  }
}
```

第一版数据类型建议：

- `BLOOD_PRESSURE`
- `BLOOD_GLUCOSE`
- `WEIGHT`
- `EXERCISE_MINUTES`
- `SLEEP_HOURS`

后端校验规则：

- `BLOOD_PRESSURE`：格式必须为 `收缩压/舒张压`，如 `120/80`；收缩压范围 `40-300`，舒张压范围 `30-200`，且收缩压必须大于舒张压。
- `BLOOD_GLUCOSE`：必须为 `0.1-50` 之间的数字。
- `WEIGHT`：必须为 `1-500` 之间的数字。
- `EXERCISE_MINUTES`：必须为 `0-1440` 之间的整数。
- `SLEEP_HOURS`：必须为 `0-24` 之间的数字。
- `unit` 若传入，必须与 `dataType` 匹配；不匹配时请求失败。
- `recordTime` 不能晚于服务端当前时间。

单位规则：

- `BLOOD_PRESSURE` → `mmHg`
- `BLOOD_GLUCOSE` → `mmol/L`
- `WEIGHT` → `kg`
- `EXERCISE_MINUTES` → `分钟`
- `SLEEP_HOURS` → `小时`

重复数据规则：

- 以“当前用户 + `dataType` + `recordTime`”判定是否重复。
- 当存在重复且 `overwrite=false` 时，接口返回 `40901`，并附带已有数据与当前提交数据供前端确认。
- 当前端确认覆盖后，使用同一接口并传 `overwrite=true` 即可完成覆盖。

重复冲突示例：

```json
{
  "code": 40901,
  "message": "该时间已存在同类型健康数据，请确认是否覆盖原记录",
  "data": {
    "existingRecord": {
      "dataType": "WEIGHT",
      "dataValue": "71.0",
      "unit": "kg",
      "recordTime": "2026-05-24 08:30:00"
    },
    "incomingRecord": {
      "dataType": "WEIGHT",
      "dataValue": "70.5",
      "unit": "kg",
      "recordTime": "2026-05-24 08:30:00"
    }
  }
}
```

错误示例：

```json
{
  "code": 40000,
  "message": "BLOOD_PRESSURE格式必须为收缩压/舒张压，例如120/80",
  "data": null
}
```

### 3.2 查询健康数据

```http
GET /api/health-data?dataType=WEIGHT&startTime=2026-05-01 00:00:00&endTime=2026-05-24 23:59:59&page=1&size=10
```

响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "items": [
      {
        "dataId": 1,
        "dataType": "WEIGHT",
        "dataValue": "70.5",
        "recordTime": "2026-05-24 08:30:00"
      }
    ],
    "page": 1,
    "size": 10,
    "total": 1
  }
}
```

说明：

- 支持按 `dataType`、`startTime`、`endTime` 筛选。
- 返回结果默认按 `recordTime` 倒序排序；若时间相同，再按 `dataId` 倒序排序。

### 3.3 删除健康数据

```http
DELETE /api/health-data/{dataId}
```

规则：

- 普通用户只能删除自己的数据。
- 管理员是否允许删除用户健康数据，后续评审决定。

### 3.4 批量导入健康数据

```http
POST /api/health-data/import
Content-Type: multipart/form-data
```

表单字段：

- `file`: CSV 文件。
- `duplicateAction`：可选，`OVERWRITE` 或 `SKIP`，用于处理导入时发现的重复数据。

响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "importedCount": 5,
    "overwrittenCount": 0,
    "skippedDuplicateCount": 0
  }
}
```

说明：

- CSV 模板当前由前端静态文件 `/health_data_template.csv` 提供。
- CSV 表头固定为 `dataType,dataValue,unit,recordTime`。
- `recordTime` 格式固定为 `yyyy-MM-dd HH:mm:ss`。
- `dataType`、`dataValue`、`unit`、`recordTime` 的业务校验规则与单条录入接口一致。
- `unit` 为空时由后端按 `dataType` 自动补默认单位。
- 任一数据行格式错误时整文件导入失败，不写入部分数据。
- 对同一 CSV 文件内重复的 `dataType + recordTime`，后出现的记录覆盖前一条待导入记录。
- 当导入内容与数据库中已有数据重复且未传 `duplicateAction` 时，接口返回 `40902`，由前端决定“覆盖全部”或“跳过重复”。

重复冲突示例：

```json
{
  "code": 40902,
  "message": "CSV中存在重复健康数据，请选择覆盖或跳过重复项",
  "data": {
    "duplicateCount": 2,
    "duplicates": [
      {
        "existingRecord": {
          "dataType": "WEIGHT",
          "dataValue": "71.0",
          "unit": "kg",
          "recordTime": "2026-05-24 08:30:00"
        },
        "incomingRecord": {
          "dataType": "WEIGHT",
          "dataValue": "70.5",
          "unit": "kg",
          "recordTime": "2026-05-24 08:30:00"
        }
      }
    ]
  }
}
```

## 8. 管理员接口

管理员接口要求用户角色为 `ADMIN`。

第一阶段不提供公开的管理员注册接口。管理员账号由初始化 SQL 创建；后续如果需要新增管理员账号，应通过受保护的管理员接口实现。

### 8.1 用户列表

```http
GET /api/admin/users?keyword=138&page=1&size=10
```

### 8.2 启用或禁用用户

```http
PATCH /api/admin/users/{userId}/status
```

请求：

```json
{
  "status": "DISABLED"
}
```

### 8.3 平台统计

```http
GET /api/admin/statistics?startDate=2026-05-01&endDate=2026-05-24
```

响应字段建议：

- 用户总数。
- 健康数据总数。
- 风险等级分布。
- 每日数据录入量。

### 8.4 登录日志查询

```http
GET /api/admin/login-logs?userId=1&result=1&page=1&size=10
```

响应字段建议：

- `logId`
- `userId`
- `phone`
- `loginIp`
- `loginTime`
- `loginResult`

### 8.5 健康知识管理

查询：

```http
GET /api/admin/knowledge?keyword=睡眠&page=1&size=10
```

新增：

```http
POST /api/admin/knowledge
```

```json
{
  "title": "睡眠健康建议",
  "content": "保持规律作息，睡前减少电子屏幕使用。",
  "relateTag": "睡眠不足",
  "source": "系统内置"
}
```

修改：

```http
PUT /api/admin/knowledge/{knowledgeId}
```

删除：

```http
DELETE /api/admin/knowledge/{knowledgeId}
```

规则：

- 健康知识管理接口仅管理员可访问。
- 如果知识已经被推送，删除策略后续评审决定；第一版建议软删除或禁止删除。
