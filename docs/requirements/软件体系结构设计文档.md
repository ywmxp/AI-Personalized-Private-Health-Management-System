## 2. 系统总体描述

### 2.1 产品愿景

打造轻量化、易操作、合规化的Web端健康管理工具，采用前后端分离架构，整合AI服务能力，为普通用户提供一站式个性化健康管理服务。

### 2.2 用户群体

#### 2.2.1 普通用户（核心用户）

- 特征：18-65岁普通/亚健康人群，无专业医疗背景；

- 核心诉求：健康数据管理、AI健康分析、个性化健康指导、健康提醒。

#### 2.2.2 管理员

- 特征：系统维护人员；

- 核心诉求：用户管理、平台数据统计、系统运行监控、健康知识维护。

### 2.3 架构设计概述

系统采用**前后端分离+四层分层架构**，基于高内聚、低耦合原则，将系统划分为**前端层、后端层、AI服务层、数据层**，各层职责独立、接口标准化，支持独立开发、测试与部署，适配课程技术栈与5周开发工期。

### 2.4 运行环境

- 客户端：Windows 10及以上，Chrome/Firefox/Edge主流浏览器；

- 服务端：Spring Boot 3.5.7、Vue 3.5、FastAPI 0.95+、MySQL 8.4；

- 依赖环境：JDK 1.8+、Python 3.9+、Node.js 24+；

- 部署环境：本地Windows环境，支持单机部署，无需集群配置。

## 3. 系统总体架构设计

### 3.1 架构模式

系统采用**四层分层架构模式**，结合**微服务轻量化设计**，前端与后端完全解耦，业务服务与AI服务独立部署，数据层集中管理，确保架构简洁、清晰、易实现。

### 3.2 系统分层架构图（UML）

```Plain Text
graph TD
    A[前端层<br/>Vue] -->|HTTP/JSON| B
    B[后端层<br/>Spring Boot] -->|HTTP/JSON| C
    B -->|JDBC| D
    C[AI服务层<br/>FastAPI] -->|调用| E[公开大模型]
    D[数据层<br/>MySQL]
```

### 3.3 架构设计特点

1. **分层独立**：四层职责清晰，层间仅通过标准化接口通信，互不干扰；

2. **前后端解耦**：前端专注交互展示，后端专注业务逻辑，支持并行开发；

3. **AI服务独立**：AI能力封装为独立服务，便于接口替换、模型升级；

4. **轻量化实现**：无复杂分布式组件，单机部署即可运行；

5. **高可维护性**：代码模块化、接口标准化，便于后期功能扩展与问题排查。

## 4. 分层详细设计

### 4.1 前端层（Vue）

- **核心职责**：用户交互、页面渲染、数据展示、请求转发、可视化呈现；

- **技术选型**：Vue 3.5；

- **核心模块**：用户登录/注册模块、健康数据管理模块、AI健康分析模块、健康提醒模块、管理员后台模块；

- **交互逻辑**：接收用户操作→发送HTTP请求至后端层→接收响应数据→渲染页面展示。

### 4.2 后端层（Spring Boot）

- **核心职责**：业务逻辑处理、权限控制、数据校验、接口转发、定时任务调度、日志管理；

- **技术选型**：Spring Boot 3.5.7；

- **核心模块**：用户管理模块、健康数据处理模块、AI接口代理模块、提醒管理模块、管理员业务模块；

- **交互逻辑**：接收前端请求→校验权限与参数→处理业务逻辑→调用AI服务/数据层接口→封装响应结果返回前端。

### 4.3 AI服务层（FastAPI）

- **核心职责**：健康画像生成、个性化健康计划生成、健康知识匹配、大模型API对接；

- **技术选型**：FastAPI 0.95+、Python 3.9+、Requests（接口调用）；

- **核心接口**：健康画像生成接口、个性化健康计划接口、健康知识推送接口；

- **交互逻辑**：接收后端层请求→解析用户健康数据→调用公开大模型API→生成分析结果→封装JSON响应返回后端层。

### 4.4 数据层（MySQL）

- **核心职责**：数据存储、数据持久化、数据查询、数据更新、数据备份；

- **技术选型**：MySQL 8.4、JDBC、MyBatis；

- **核心数据表**：用户表、健康数据表、健康画像表、健康计划表、提醒表、健康知识库表、用户反馈表、登录日志表；

- **交互逻辑**：接收后端层数据操作请求→执行SQL语句→返回查询结果/操作状态→持久化数据。

- **低耦合**：模块间依赖最小化，通过接口通信，无直接代码依赖。

## 6. 组件交互设计

### 6.1 核心组件划分

系统核心组件按功能划分为**用户交互组件、业务逻辑组件、AI服务组件、数据访问组件、可视化组件**，各组件独立封装、可复用。

### 6.2 核心组件交互图（UML）

```Plain Text
graph TD
    C1[用户交互组件<br/>Vue页面] -->|HTTP请求| C2[业务逻辑组件<br/>Spring Boot]
    C2 -->|HTTP请求| C3[AI服务组件<br/>FastAPI]
    C2 -->|SQL操作| C4[数据访问组件<br/>MyBatis]
    C4 -->|读写数据| C5[数据库组件<br/>MySQL]
    C1 -->|渲染数据| C6[可视化组件<br/>ECharts]
```

### 6.3 核心交互流程

1. **用户登录交互**：用户交互组件→业务逻辑组件→数据访问组件→数据库组件→返回用户信息→用户交互组件渲染页面；

2. **健康画像生成交互**：用户交互组件→业务逻辑组件→AI服务组件→调用大模型→返回画像结果→业务逻辑组件→数据访问组件→数据库组件→返回结果→用户交互组件渲染画像；

3. **健康数据录入交互**：用户交互组件→业务逻辑组件→数据访问组件→数据库组件→返回保存结果→用户交互组件提示成功。

## 7. 接口设计

### 7.1 层间接口设计

#### 7.1.1 前端层-后端层接口

- **协议**：HTTP/HTTPS；

- **请求方式**：GET（查询）、POST（新增/提交）、PUT（修改）、DELETE（删除）；

- **数据格式**：JSON；

- **核心接口**：用户注册/登录接口、健康数据录入/查询接口、AI健康画像接口、提醒设置接口、管理员管理接口。

#### 7.1.2 后端层-AI服务层接口

- **协议**：HTTP；

- **请求方式**：POST；

- **数据格式**：JSON；

- **接口地址**：[http://localhost:8000/api/ai/xxx；](http://localhost:8000/api/ai/xxx；)

- **核心接口**：健康画像生成接口、个性化健康计划接口、健康知识推送接口。

#### 7.1.3 后端层-数据层接口

- **协议**：JDBC；

- **数据格式**：SQL语句、ResultSet；

- **核心接口**：用户数据CRUD接口、健康数据CRUD接口、AI结果数据CRUD接口、提醒数据CRUD接口。

### 7.2 接口设计规范

- 接口命名：简洁清晰，采用“模块_功能”格式（如user_register、health_data_input）；

- 请求参数：必填参数非空校验，参数类型严格匹配；

- 返回参数：统一JSON格式，包含状态码、提示信息、数据体；

- 异常处理：接口调用失败返回友好提示，不暴露系统底层错误。

## 8. 部署设计

### 8.1 部署架构

系统采用**单机部署架构**，所有服务（前端、后端、AI服务、数据库）部署在同一台Windows服务器上，通过本地端口区分服务，简化部署流程。

### 8.2 系统部署图（UML）

```Plain Text
graph TD
    Client[用户浏览器<br/>Chrome/Firefox] -->|访问| Server[Windows服务器<br/>Windows 10+]
    Server --> FrontEnd[前端服务<br/>Vue:8080]
    Server --> BackEnd[后端服务<br/>Spring Boot:8081]
    Server --> AIService[AI服务<br/>FastAPI:8000]
    Server --> Database[数据库<br/>MySQL:3306]

    FrontEnd --> BackEnd
    BackEnd --> AIService
    BackEnd --> Database
```

### 8.3 部署流程

1. 部署MySQL数据库：安装MySQL 8.4，创建系统数据库，导入数据表结构；

2. 部署AI服务：安装Python 3.9+，安装依赖包，启动FastAPI服务（端口8000）；

3. 部署后端服务：安装JDK 1.8+，打包Spring Boot项目，启动服务（端口8081）；

4. 部署前端服务：安装Node.js 24+，打包Vue项目，部署至Nginx（端口8080）；

5. 系统联调：验证各服务通信正常，核心功能可正常访问。

## 9. 设计约束

### 9.1 技术约束

- 必须采用**Spring Boot+Vue+FastAPI+MySQL**核心技术栈，禁止替换核心框架；

- AI服务仅对接公开大模型API，不自主训练医疗模型；

- 前端仅适配PC端浏览器，不开发移动端适配页面。

### 9.2 资源约束

- 开发资源：9人团队，分前端、后端、AI开发、测试小组；

- 工期约束：5周内完成架构实现、编码开发、集成测试、部署演示；

- 硬件资源：普通PC机即可满足开发与部署需求，无需高配服务器。

### 9.3 合规约束

- AI服务输出内容仅提供生活方式指导，**不涉及医疗诊断、用药推荐**；

- 健康知识内容符合权威健康规范，无虚假、违规信息；

- 用户数据仅存储于本地数据库，不泄露、不对外共享，符合隐私保护要求。
